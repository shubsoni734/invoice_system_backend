## PERMISSION MODEL — THREE TIERS

```
TIER 1: SuperAdmin (Platform Owner)
  Roles: superadmin | support | finance | readonly
  Scope: ALL organisations, ALL data
  Auth:  POST /superadmin/auth/login  →  separate JWT issuer SA_JWT_PRIVATE_KEY

TIER 2: Organisation Admin
  Roles: admin | manager | viewer
  Scope: OWN organisation ONLY — all queries MUST filter by organisation_id
  Auth:  POST /api/v1/auth/login  →  JWT issuer ORG_JWT_PRIVATE_KEY

TIER 3: (Future) Organisation Member
  Roles: staff
  Scope: Limited operations within own organisation
```

> **CRITICAL RULE:** SuperAdmin JWTs and Org JWTs use DIFFERENT private keys and DIFFERENT middleware chains. They must never be interchangeable.

## ALL API ROUTES

### SuperAdmin Routes — prefix: `/superadmin`
> Protected by `SuperAuthMiddleware` + `SuperRBACMiddleware`

| Method | Route | Role Required | Description |
|--------|-------|--------------|-------------|
| POST | `/superadmin/auth/login` | Public | SuperAdmin login |

## SPECIAL BUSINESS LOGIC — IMPLEMENT EXACTLY AS DESCRIBED

### 1. Invoice Number Generation — ATOMIC
```sql
-- Use this query in sqlc — atomic, no race condition
-- File: internal/db/queries/invoice_sessions.sql

-- name: IncrementAndGetSequence :one
UPDATE invoice_sessions
SET current_sequence = current_sequence + 1
WHERE organisation_id = $1
  AND year = $2
  AND prefix = $3
RETURNING current_sequence, prefix, year;
```
Format result as: `{prefix}-{year}-{sequence zero-padded to 4 digits}`
Example: `INV-2025-0001`

If no session exists for the year, auto-create one inside a transaction before incrementing.

---

### 2. Service Auto-Fill on Invoice Item Creation
When `service_id` is provided in an invoice item:
1. Fetch the service record
2. Copy `unit_price`, `description`, and `tax_rate` into the invoice item fields
3. These values are **copied, not referenced** — historical accuracy is preserved if the service price changes later
4. The user may override copied values before saving

---

### 3. Auto Customer Creation on Invoice Creation
```
If request contains customer_id → use it directly
If request contains customer object (no customer_id):
  → Search by (organisation_id + phone) OR (organisation_id + email)
  → If found: use existing customer_id
  → If not found: INSERT new customer within the same transaction
  → Return resolved customer_id in invoice response
```

---

### 4. Invoice Total Calculation
```
For each item:
  tax_amount = quantity × unit_price × (tax_rate / 100)
  line_total  = (quantity × unit_price) + tax_amount

Invoice:
  subtotal        = SUM(quantity × unit_price for all items)
  tax_amount      = SUM(tax_amount for all items)
  discount_amount = applied discount (if any)
  total           = subtotal + tax_amount - discount_amount
```
All calculations done in the service layer. Store results in DB.

---

### 5. Payment — Auto Invoice Status Update
After recording a payment:
```
total_paid = SUM(amount) FROM payments WHERE invoice_id = X
if total_paid >= invoice.total → update invoice status to 'paid', set paid_at = NOW()
if total_paid > 0 AND total_paid < invoice.total → status remains 'sent' (partial)
```

---

### 6. WhatsApp — Async Sending
- WhatsApp messages are sent **asynchronously via a goroutine** — never block the HTTP response
- Insert a record into `whatsapp_logs` with status `pending` before spawning the goroutine
- The goroutine updates the log record to `sent` or `failed` after the API call completes
- If `whatsapp_enabled = false` in settings, return a 422 with `"WhatsApp is not enabled for this organisation"`

---

### 7. Plan Limit Enforcement
Check in `PlanLimitMiddleware` before any CREATE operation:
```
Before creating an invoice:
  count = SELECT COUNT(*) FROM invoices WHERE organisation_id = X
          AND created_at >= date_trunc('month', NOW())
  if count >= plan.max_invoices_per_month → return 403 ErrPlanLimit

Before creating a user:
  count = SELECT COUNT(*) FROM users WHERE organisation_id = X AND is_active = TRUE
  if count >= plan.max_users → return 403 ErrPlanLimit

Before creating a customer:
  count = SELECT COUNT(*) FROM customers WHERE organisation_id = X AND is_active = TRUE
  if count >= plan.max_customers → return 403 ErrPlanLimit
```

---

### 8. Impersonation Flow
```
SuperAdmin POSTs to /superadmin/organisations/:id/impersonate
  Body: { "target_user_id": 5, "reason": "Customer support request #1234" }

1. Verify requesting super_admin has role superadmin or support
2. Verify target organisation is active
3. Insert into impersonation_sessions with reason + token_hash
4. Issue a short-lived (1 hour) Org JWT with:
   - Standard org user claims
   - Extra claim: "impersonated_by": super_admin_id
   - Extra claim: "impersonation_session_id": session_id
5. Return the impersonation JWT
6. All actions taken with this token are logged with actor_type = 'superadmin' in audit_logs
```

## SECURITY — NON-NEGOTIABLE REQUIREMENTS

```
JWT:
  - Use RS256 (asymmetric) — NEVER HS256
  - Two separate key pairs: SA_JWT_PRIVATE_KEY and ORG_JWT_PRIVATE_KEY
  - Access token expiry: 15 minutes
  - Refresh token expiry: 7 days
  - Store refresh token as SHA-256 hash in DB — never plaintext

Passwords:
  - Hash with bcrypt, cost 12
  - Minimum 8 characters enforced at validation layer
  - Account lockout: 5 failed attempts → locked for 15 minutes

Rate Limiting:
  - /auth/login and /superadmin/auth/login: 10 requests/minute per IP
  - All other endpoints: 300 requests/minute per user
  - Return 429 with Retry-After header

HTTP Security Headers (set in SecurityHeadersMiddleware):
  - Strict-Transport-Security: max-age=31536000; includeSubDomains
  - X-Content-Type-Options: nosniff
  - X-Frame-Options: DENY
  - X-Request-ID: {uuid}
  - Referrer-Policy: no-referrer
  - Permissions-Policy: camera=(), microphone=()

CORS:
  - Never use wildcard *
  - Read allowed origins from config/env
  - Only allow configured origins, methods, and headers

Input:
  - Max request body size: 1MB (2MB for file uploads)
  - Validate ALL fields using go-playground/validator
  - Whitelist file upload types: .png .jpg .webp for logos
  - Never expose raw DB errors to the client

SuperAdmin extra:
  - Mandatory MFA (TOTP) on all SuperAdmin accounts
  - IP allowlist for SuperAdmin login (read from config)
  - All mutations logged to super_audit_logs
  - Impersonation always requires a written reason
```
